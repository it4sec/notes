# Analysis of Security Vulnerabilities in S-100-Based Maritime Navigation Software

**Authors:** Hoyeon Cho, Changui Lee, and Seojeong Lee  
**Journal:** Sensors (2026), 26, 1246  
**DOI:** https://doi.org/10.3390/s26041246  
**Date Read:** 2026-06-20

## Abstract in One Paragraph

This paper presents one of the first empirical cybersecurity evaluations of S-100-based maritime navigation software, focusing on the security implications of Lua scripting within next-generation Electronic Chart Display and Information Systems (ECDIS). Using OpenS100 as a reference implementation, the authors compare four automated Static Application Security Testing (SAST) tools against expert manual review. Although automated tools detected 1,530 unique defects, they missed 83% of expert-identified vulnerabilities. The most critical issue was an unrestricted Lua interpreter that enabled Remote Code Execution (RCE) through malicious Portrayal Catalogues (PCs), demonstrating that current S-100 specifications lack essential security requirements for script execution environments.

---

## Research Problem

The S-100 standard mandates Lua 5.1 for chart portrayal but does not define:

- Script sandboxing requirements
- Library restrictions
- Capability controls
- Secure execution boundaries

As a result, standards-compliant implementations may introduce severe vulnerabilities even when they fully conform to functional requirements.

---

## Research Questions

1. How effective are general-purpose SAST tools at detecting vulnerabilities in S-100 software?
2. What security risks emerge from the C++–Lua trust boundary?
3. Are identified vulnerabilities implementation bugs or consequences of specification-level design gaps?
4. How should future S-100 security requirements evolve before mandatory adoption in 2029?

- First empirical security assessment of S-100 portrayal engines.
- First quantitative comparison between SAST tools and expert review for maritime navigation software.
- Demonstrates that 83% of critical vulnerabilities remain undetected by conventional tooling.
- Identifies a specification-level vulnerability rather than a simple coding defect.
- Provides a working proof-of-concept for RCE through malicious Portrayal Catalogues.
- Shows that functional safety certification alone is insufficient for maritime cybersecurity.
- Proposes concrete mitigation strategies for future S-100 revisions.

## Why Existing SAST Tools Failed

The paper identifies four root causes:

1. **No maritime domain knowledge**
2. **Inability to analyze C++–Lua interactions**
3. **Lack of supply-chain threat modeling**
4. **No visibility into runtime interpreter configuration**

Examples of missed vulnerabilities:

- Lua sandbox failures
- Path traversal
- SQL injection
- Cross-file trust boundary violations
- Dynamic script execution risks

---

## Case Study: Unrestricted Lua Interpreter

### Root Cause

OpenS100 initializes Lua using:

```cpp
lua_State* L = luaL_newstate();
luaL_openlibs(L);
```

This exposes dangerous libraries:

- `os`
- `io`
- `debug`
- `package`
- `loadfile`
- `loadstring`

As a result, malicious PCs can execute arbitrary operating system commands.

### Attack Chain

1. Attacker modifies a Portrayal Catalogue.
2. Compromised catalogue is distributed through legitimate channels.
3. User loads chart data.
4. Lua script executes automatically.
5. Arbitrary code runs on the host system.

### Why This Matters

- Exploitation requires no memory corruption.
- DEP and ASLR provide little protection.
- Malicious activity is invisible to navigators.
- Charts continue rendering normally.
- Centralized distribution creates supply-chain risk.

## Practical Examples for Researchers

### Example 1 — Supply-Chain Compromise

A compromised hydrographic office distributes a signed but malicious Portrayal Catalogue.

Research directions:

- Secure update pipelines
- Provenance verification
- Reproducible catalogue builds

### Example 2 — Cross-Language Analysis

Critical vulnerabilities emerged at the C++–Lua boundary.

Research directions:

- Cross-language taint analysis
- Hybrid static/dynamic analysis
- Semantic modeling across runtimes

### Example 3 — Safety–Security Convergence

Chart manipulation can lead directly to navigation hazards.

Research directions:

- Joint safety-security assurance
- Maritime threat modeling
- Functional safety extensions

---

## Recommended Mitigations

### Immediate Controls

- Disable dangerous Lua libraries
- Enforce interpreter sandboxing
- Restrict native library loading
- Implement capability whitelists
- Validate Portrayal Catalogue integrity

### Standards-Level Changes

The authors recommend that IHO:

- Add mandatory sandbox requirements to Part 13
- Require security controls in Part 9a
- Make PC signatures mandatory
- Define secure callback interfaces
- Include security requirements in certification programs

---

## Research Implications

This work demonstrates that:

- Standards can introduce vulnerabilities even when implementations are correct.
- Domain-specific security expertise remains essential.
- General-purpose SAST tools require maritime extensions.
- Security validation must complement functional certification.
- Future maritime software assurance should integrate safety, security, and supply-chain analysis.

---

## Key Takeaways

1. S-100 introduces a new attack surface through executable portrayal logic.
2. Authenticity of chart data does not guarantee safety of chart scripts.
3. Manual review remains essential for safety-critical systems.
4. Cross-language interfaces are a major source of hidden vulnerabilities.
5. Security requirements must be embedded directly into international standards.
